# Typora 与 Typedown 功能总结

> 文档日期：2026-05-29  
> 适用项目：一个轻量化 Markdown 阅读编辑器 / 编辑器内核选型参考  
> 资料边界：Typora 以官方功能页为主；Typedown 以 `byxiaozhi/Typedown` 开源仓库、Microsoft Store 页面，以及你提供的截图观察为依据。Typedown 的公开功能描述明显少于 Typora，因此截图观察项在本文中按“可观察 / 可推断”处理。

---

## 1. 总体结论

Typora 和 Typedown 都不是单纯的“源码编辑器”，而是偏向 **Markdown 文件 + 所见即所得 / 所见即语义** 的轻量写作工具。

对本项目最有参考价值的不是照搬功能数量，而是抽象出三条产品原则：

1. **Markdown-first**：最终内容仍然是 `.md` 文本，而不是私有富文本文档格式。
2. **低干扰写作**：默认界面尽量少打扰用户，格式工具只在需要时出现。
3. **本地文件体验优先**：打开、编辑、保存、图片路径、导出、目录导航都要围绕本地 Markdown 文件工作流设计。

## 2. Typora 功能总结

### 2.1 产品定位

Typora 的核心定位是：**一个极简但功能完整的 Markdown 阅读与写作工具**。

它的关键体验是取消传统 Markdown 编辑器中“左边源码、右边预览”的双栏模式，改为在同一个编辑区域中实时渲染 Markdown 语义。官方将其描述为移除预览窗口、模式切换和 Markdown 源码符号干扰，提供真正的实时预览体验。[1]

### 2.2 核心功能矩阵

| 功能类别 | Typora 表现 | 对本项目的参考价值 |
| --- | --- | --- |
| 编辑体验 | 实时预览、无独立预览窗口、WYSIWYM 风格 | 第一版应提供“编辑即预览”的主体验，而不是只做源码 textarea |
| Markdown 基础语法 | 标题、列表、表格、代码块、图片、链接、引用、分割线、行内样式 | 需要作为 EditorCore 的基础 schema，而不是插件化后置功能 |
| 图片处理 | 支持图片显示、拖放插入、相对路径、图片尺寸 / zoom、部分平台图片上传 | 本项目应设计 `imageLoader`，统一处理本地路径、复制资源、预览 URL、失败占位 |
| 列表与任务 | 支持缩进 / 反缩进、GFM task list、列表类型切换 | 列表编辑行为要接近富文本编辑器，避免用户手写所有缩进 |
| 表格 | 支持快捷插入、Markdown 输入、拖拽调整表格尺寸 | MVP 可先支持基础表格，后续再做拖拽调整、列宽、对齐 |
| 代码块 | 支持代码块、行号、约 100 种语言语法高亮 | 本项目应将代码高亮抽象为 `codeTokenizer`，避免编辑器内核直接绑定 Shiki |
| 数学公式 | 支持 MathJax 扩展、公式自动编号 | 本项目可选 KaTeX，MVP 先支持行内 / 块级公式渲染，自动编号后置 |
| 图表 | 支持 Mermaid、flowchart、sequence 等文本图表 | Mermaid 应作为代码块渲染扩展，而不是直接进入核心文档模型 |
| 目录与导航 | `[TOC]`、内部链接、Outline Panel | 本项目应提供 `outlineService`，从 Markdown AST / ProseMirror doc 中派生目录 |
| 文件组织 | 文件树、文章列表侧栏、可放入 Dropbox / iCloud 等同步目录 | 桌面端可优先做“文件夹工作区”，同步交给外部云盘即可 |
| 导入导出 | PDF 书签导出、docx、OpenOffice、LaTeX、MediaWiki、Epub 等格式 | MVP 不宜做全量导出，建议先 PDF / HTML / Markdown 原文保存 |
| 写作辅助 | 字数统计、阅读时间、专注模式、打字机模式、自动配对括号 / 引号 | 字数统计和专注模式成本低、感知强，适合第一版加入 |
| 主题系统 | 支持自定义 CSS | 本项目应从一开始设计 CSS 变量，而不是硬编码主题样式 |
| CJK 支持 | 中日韩字符下的加粗、斜体等样式正确渲染 | 中文用户场景下必须专项测试 Markdown 标记和 IME 输入 |

### 2.3 Typora 的体验重点

#### 2.3.1 “源码符号只在需要时出现”

Typora 的优势不是完全隐藏 Markdown，而是让用户大部分时间关注内容本身。用户在标题、列表、表格、代码块、公式等区域写作时，看到的是语义化后的文档形态。

本项目不必完全复刻 Typora，但应追求：

- 普通段落中减少 Markdown 符号干扰。
- 光标进入某个节点时，可以显示必要的 Markdown 语义边界。
- 保存时必须保留 Markdown 文本作为最终结果。

#### 2.3.2 图片体验是 Markdown 编辑器的分水岭

Typora 明确支持拖放图片、相对路径、图片尺寸等能力。[1] 这说明 Markdown 编辑器不能只处理文本，还必须解决“图片文件如何进入 Markdown 文档”的问题。

本项目应抽象：

```ts
export interface ImageLoader {
  resolve(src: string, context: ImageContext): Promise<ImageResolveResult>;
  import(file: File | string, context: ImageContext): Promise<ImageImportResult>;
}
```

这样后续可以同时支持：

- 本地相对路径图片；
- 粘贴图片；
- 拖放图片；
- 复制到文档资源目录；
- 未来可选的图床 / 云同步。

#### 2.3.3 表格、列表、代码块决定“像不像编辑器”

Typora 对列表、任务、表格、代码块做了大量编辑体验优化。[1] 这类能力是 Markdown 编辑器最容易拉开差距的地方。

本项目第一版需要优先做好：

1. Enter / Backspace 在列表中的行为；
2. Tab / Shift+Tab 缩进；
3. 代码块语言选择与高亮；
4. 表格的基础插入、增删行列；
5. 任务列表 checkbox 与 Markdown `- [ ]` / `- [x]` 同步。

## 3. Typedown 功能总结

### 3.1 产品定位

本文中的 Typedown 指 `byxiaozhi/Typedown`，它在 GitHub 上描述为：**专门为 Windows 平台设计的轻量 Markdown 编辑器，使用 WinUI 框架，目标是提供贴合操作系统的界面与高效编辑体验**。[2]

它相对于 Typora 的参考价值主要在于：

- 更接近 Windows 原生应用审美；
- 功能表达更轻量；
- 适合作为轻量桌面 Markdown 编辑器的 UI 参考；
- 开源项目结构可以观察其桌面壳与 Web 编辑器资源的组合方式。

### 3.2 可确定功能

| 功能类别 | Typedown 表现 | 依据 | 对本项目的参考价值 |
| --- | --- | --- | --- |
| 平台定位 | Windows 平台轻量 Markdown 编辑器 | GitHub / Microsoft Store 描述 | 若本项目使用 Tauri 2，要尽量贴合系统窗口、菜单、快捷键习惯 |
| UI 技术 | WinUI 框架 | GitHub / Microsoft Store 描述 | Tauri + Svelte 也应提供接近原生的窗口和交互密度 |
| Markdown 编辑 | 面向技术文档、学术论文、博客写作 | GitHub README 描述 | 本项目应把“普通写作 + 技术文档”作为主场景 |
| 开源属性 | GitHub 开源，MIT license | GitHub 仓库 | 可作为竞品学习对象，但不代表可直接复用其全部实现 |
| 构建结构 | 包含 `Typedown.Editor` 前端资源，再嵌入桌面项目 | GitHub 构建说明 | 与 Tauri 的“前端 UI + 桌面壳”模式有相似工程边界 |

### 3.3 从截图可观察 / 可推断的功能方向

> 说明：以下来自你提供的 Typedown 截图观察和同类编辑器常见交互归纳，需以你实际安装版本为准。

| 功能方向 | 可观察 / 可推断内容 | 对本项目的参考价值 |
| --- | --- | --- |
| 简洁工具栏 | 顶部或局部工具栏承载常用格式操作 | 第一版可以提供轻量 toolbar，但不要压过写作区 |
| 分栏 / 预览 | 可能支持编辑与预览相关布局 | 本项目应明确主体验是“Markdown-first 所见即语义”，源码模式作为可切换模式 |
| 主题与暗色模式 | 截图中可见现代化主题倾向 | CSS 变量要覆盖编辑区、代码块、引用、表格、浮层、选区 |
| 文档区域聚焦 | 编辑区域相对干净 | 不宜第一版加入复杂知识库、账户、云同步等大系统 |
| Windows 原生感 | 标题栏、控件风格、窗口行为接近 Windows 应用 | Tauri 端要处理窗口菜单、文件拖放、最近文件、系统对话框 |

### 3.4 Typedown 的产品启发

Typedown 的启发不是“功能要比 Typora 多”，而是：**轻量桌面应用可以靠平台贴合度和低复杂度获得好体验**。

对本项目来说，Typedown 更适合作为 UI/交互参考：

- 窗口简洁；
- 快速打开本地 `.md`；
- 不强迫用户进入复杂工作区；
- 常用编辑能力触手可及；
- 系统暗色 / 浅色主题自然切换。

## 4. Typora 与 Typedown 对比

| 维度 | Typora | Typedown | 本项目建议 |
| --- | --- | --- | --- |
| 产品成熟度 | 成熟商业产品 | 轻量开源 Windows 应用 | MVP 先接近 Typedown 的轻量，长期学习 Typora 的细节深度 |
| 核心体验 | 单窗口实时预览 / WYSIWYM | Windows 原生轻量编辑 | 主体验采用 Markdown-first 所见即语义 |
| 平台覆盖 | macOS / Windows / Linux | Windows 为主 | Tauri 2 可跨平台，但第一版优先保证 Windows / macOS 之一体验完整 |
| 功能深度 | 图片、表格、数学、图表、导入导出、主题都较完整 | 公开资料显示较轻量 | 第一版控制范围，避免一开始做完整 Typora |
| 技术参考 | 产品体验参考 | 桌面轻量 UI 参考 | 不照搬内核，提炼交互和功能优先级 |
| 文件工作流 | 本地 Markdown 文件 + 文件树 / 文章列表 | 本地 Markdown 编辑 | 本地文件系统是主存储，SQLite 只做索引 / 元数据 / 缓存 |

## 5. 对本项目的功能优先级建议

### 5.1 MVP 必做

| 优先级 | 功能 | 原因 |
| --- | --- | --- |
| P0 | 打开、编辑、保存 `.md` 文件 | Markdown-first 的根能力 |
| P0 | ProseMirror 编辑区 + Markdown Parser / Serializer | 建立稳定编辑内核 |
| P0 | 标题、段落、粗体、斜体、链接、引用、列表、代码块 | 基础 Markdown 写作必需 |
| P0 | EditorCore 封装层 | 第一版就避免 UI 与 ProseMirror 深度耦合 |
| P0 | 源码模式 / 语义编辑模式切换 | 满足 Markdown 用户对源码可控性的要求 |
| P1 | 图片拖放、粘贴、相对路径预览 | Markdown 编辑器的关键体验点 |
| P1 | 代码块 Shiki 高亮 | 技术文档场景必需 |
| P1 | Outline / 文档目录 | 长文档阅读与导航必需 |
| P1 | CSS 变量主题 | 后续主题系统和暗色模式基础 |
| P1 | 字数统计、当前行 / 选区状态 | 低成本高感知 |

### 5.2 MVP 可延后

| 功能 | 延后原因 |
| --- | --- |
| 完整导出 docx / epub / LaTeX | 工程量较大，容易偏离编辑器内核验证 |
| 复杂表格拖拽调整 | ProseMirror 表格边界复杂，先做基础表格 |
| 云同步 / 账户系统 | 与轻量本地编辑器定位冲突，后期可选 |
| AI 写作深度集成 | 应先把 EditorCore 的命令、选区、流式插入抽象做好 |
| 插件市场 | 先内部模块化，不急于开放外部插件协议 |

## 7. 最终建议

第一版不要追求“复刻 Typora 全功能”，也不要只做“Typedown 式轻量壳”。更合适的目标是：

> 做一个基于 ProseMirror 的 Svelte Markdown 编辑器，底层稳定，体验 Markdown-first，外部 API 简洁，渲染能力通过 imageLoader、codeTokenizer、CSS 变量和扩展服务逐步增强。

具体来说：

1. 产品体验学习 Typora：实时语义编辑、图片、代码块、数学、图表、目录、主题。
2. 桌面气质学习 Typedown：轻量、原生感、本地文件优先、界面克制。
3. 技术实现不要直接绑定 UI：第一版就建立 EditorCore 封装层。
4. 长期保留自研内核可能性：但第一版以 ProseMirror 降低编辑行为风险。
