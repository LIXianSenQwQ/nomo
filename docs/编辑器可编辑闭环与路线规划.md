# 编辑器可编辑闭环与路线规划

> 文档日期：2026-05-29  
> 状态：ready-for-agent  
> 输入来源：现有 `CONTEXT.md`、`docs` 产品/架构文档、`.scratch` 阶段 PRD、当前 Svelte / EditorCore / Markdown 相关源码  
> 核心结论：下一阶段必须先恢复“语义模式可编辑”和“源码模式输入稳定”这两个 P0 闭环，再继续推进渲染能力、桌面能力和长期内核边界。

## Problem Statement

项目的初衷是做一个本地优先、Markdown-first、接近 Typora 的轻量 Markdown 编辑器。现有文档已经把 **语义编辑** 定义为用户在同一编辑区中直接编辑标题、列表、引用、代码块等 Markdown 语义结构，并把 **源码模式** 定义为处理保真、调试或高级语法场景的补充形态。

当前已经发现两个直接违背核心体验的问题：

1. 语义模式下无法编辑。可见的语义区域更像阅读/预览层，而不是主编辑面。
2. 源码模式下输入文字会直接跳到最顶部。用户无法稳定连续写作。

这两个问题会让应用从“Markdown-first 语义编辑器”退化成“带预览的源码查看器”，也会让后续图片、表格、公式、Mermaid、大纲、桌面保存等能力失去可用基础。

## Current Review Findings

### 代码现状

- 应用层已经具备较多功能：桌面文件打开保存、多标签、资源管理器、文档大纲、源码 textarea、语义渲染层、技术块预览、主题设置、快照和大文件降级。
- EditorCore 已经封装 ProseMirror，并提供 Markdown 状态、命令、订阅、主题和运行时选项。
- MarkdownBridge 目前仍偏薄，主要处理 front matter 拆分和拼回；复杂 Markdown 保真、source map、raw block 还没有形成稳定模型。
- 语义模式中同时存在富 Markdown 派生渲染层和被隐藏的 ProseMirror host。富渲染层承担了用户看到的内容，但不是完整编辑器。
- 源码模式每次输入都会把 textarea 内容推入 EditorCore，再触发全量 Markdown 派生、语义解析、高度重算和 ProseMirror 状态替换。
- 已有测试覆盖 EditorCore 基础事件、front matter 保留、技术块插入、Outline 布局、richMarkdown 派生模型和部分渲染服务，但缺少真实用户输入稳定性、滚动位置、选择区、模式切换一致性测试。

### 文档现状

- `CONTEXT.md` 的领域语言是清晰的：语义编辑不是纯预览，Markdown 文本是主数据，Outline 和写作统计都是派生数据。
- `docs/PRD_轻量 Markdown 编辑器工程.md` 已经定义 0-5 阶段路线，但当前实现跳过了“语义编辑可直接写”的底层闭环，先叠加了大量语义渲染能力。
- `.scratch` 里已有阶段 PRD 和局部修复 PRD，其中 “Markdown 语义渲染补齐” 解决了阅读表现，但也暴露出下一步必须把渲染层与编辑层重新统一。
- `.scratch/tab-outline-navigation` 仍是 `needs-triage`，多标签、资源管理器和大纲联动还需要继续收敛。

## Root Cause Assessment

### P0：语义模式无法编辑

当前语义模式的可见内容来自富 Markdown 渲染模型。它能渲染标题、任务、表格、代码、数学公式和 Mermaid，但主体是普通 DOM 输出，不是 ProseMirror 的可编辑文档面。真正的 ProseMirror host 被缩小、隐藏且禁止指针交互，因此用户无法在可见语义内容上输入。

这不是单点 bug，而是产品边界偏移：**语义渲染** 被临时当成了 **语义编辑**。接下来必须把语义模式重新定义为可编辑主面，富渲染能力只能作为 ProseMirror NodeView、装饰层、只读预览子视图或局部增强存在。

### P0：源码模式输入跳到顶部

源码 textarea 每次输入都会触发全量 `setMarkdown`，随后应用层同步事件重新计算 Markdown 派生状态、重设 textarea 高度，并可能更新隐藏的 ProseMirror 状态。这个流程没有显式保存和恢复 textarea 的 selection、scrollTop 和外层滚动容器位置。高度从 `auto` 到 `scrollHeight` 的重算也容易让滚动容器回到顶部。

源码模式应成为一个稳定的原文编辑面：输入期间只更新主 Markdown 状态和轻量派生信息，不能在每个字符输入时触发会破坏滚动和光标的全量编辑器重建。

## Solution

把下一阶段目标收敛为“恢复可编辑闭环”，再逐步把已有渲染和桌面能力纳入稳定架构。

产品上，语义模式必须是默认主编辑面，用户能直接在标题、段落、列表、引用、代码块等语义结构中输入、删除、粘贴、撤销重做。源码模式是同一 Markdown 文档的原文编辑面，必须保持输入、光标、滚动和内容同步稳定。

工程上，以 EditorCore 作为唯一编辑状态边界，明确区分四类能力：

- 主数据：Markdown 文本。
- 编辑状态：EditorCore / ProseMirror 文档、选区、历史。
- 派生视图：Outline、统计、代码高亮、数学公式、Mermaid、表格展示。
- 桌面状态：路径、最近文件、快照、窗口状态、外部文件变化。

## User Stories

1. As a Markdown writer, I want semantic mode to be directly editable, so that I can write without switching back to source mode.
2. As a Markdown writer, I want headings to remain editable as headings, so that document structure is visible while writing.
3. As a Markdown writer, I want paragraphs, quotes and lists to accept normal typing, so that semantic mode is a real writing surface.
4. As a Chinese writer, I want IME composition to stay stable in semantic mode, so that Chinese input does not lose characters or move the caret.
5. As a Markdown power user, I want source mode typing to keep caret and scroll position, so that long documents can be edited safely.
6. As a technical writer, I want source and semantic mode to show the same Markdown content, so that switching modes does not fork the document.
7. As a technical writer, I want task list checkbox changes to write back to Markdown, so that rendered interactions remain Markdown-first.
8. As a technical writer, I want code, math and Mermaid blocks to keep their source text editable, so that advanced syntax can be fixed without data loss.
9. As a long-form writer, I want Outline navigation to keep my current mode, so that navigation does not disrupt editing context.
10. As a desktop user, I want save failures and external file changes to preserve my current edits, so that local files remain trustworthy.
11. As a maintainer, I want UI code to depend on EditorCore contracts rather than ProseMirror internals, so that the editing engine remains replaceable.
12. As a maintainer, I want regression tests for mode switching, scroll preservation and Markdown round trips, so that the same core bugs do not return.

## Implementation Decisions

- Treat “语义模式可编辑” as the first acceptance gate. A semantic surface that only renders Markdown is not sufficient.
- Keep one Markdown source of truth. Source mode and semantic mode must never maintain independent document copies.
- Move rich Markdown rendering from a standalone visible document into editable editor capabilities: ProseMirror schema nodes, NodeViews, decorations, or focused read-only previews inside editable blocks.
- Source mode input should avoid rebuilding the semantic editor on every keystroke unless necessary. Use scheduled synchronization, explicit mode-switch synchronization, or state-preserving updates.
- Preserve selection and scroll whenever source input causes derived state refresh.
- Stop applying whitespace-normalizing helpers on every source keystroke. Trailing blank-line normalization should run at controlled boundaries such as save, mode switch, or snippet insertion.
- Keep Outline and stats as derived state from Markdown. They can update during typing, but their update path must not force editor remounts or scroll resets.
- Keep desktop persistence behind storage services. UI should receive explicit success/failure state and should not silently discard edits after failed save.
- Record every high-risk editor behavior as an observable contract: Markdown in/out, mode switch, caret stability, scroll stability, dirty state, save recovery, and front matter preservation.

## Testing Decisions

- Add focused regression tests for the two known bugs before implementation.
- Use behavior-level tests instead of ProseMirror internals where possible.
- Add source mode tests for input preserving textarea selection and outer scroll position.
- Add mode-switch tests for semantic -> source -> semantic content equality.
- Add semantic edit tests for basic heading, paragraph, list and quote editing.
- Add IME manual verification checklist for Windows Chinese input after semantic mode changes.
- Keep Markdown fixture tests for front matter, code fences, task lists, tables, math and Mermaid.
- Run `pnpm check`, `pnpm test` and `pnpm build` before treating each phase as complete.
- For Tauri phases, also run Rust/Tauri checks when the local Rust toolchain is available.

## Phased Roadmap

### 阶段 0：止血与验收口径重置

目标：把项目重新拉回“可写编辑器”主线。

主要工作：

- 明确术语：语义模式必须可编辑；语义渲染只是语义编辑的显示增强。同时要保留当前的渲染，做到所见即所得
- 梳理当前 App 状态流：Markdown、EditorCore、textarea、richBlocks、Outline、stats、technical previews、tabs。
- 给两个已知问题建立最小复现和验收清单。
- 标记现有富渲染层的职责：哪些可迁移到 NodeView，哪些只能保留为只读预览，哪些应先回退。
- 冻结无关新增功能，避免在 P0 编辑闭环未恢复前继续扩展导出、插件、AI 或复杂表格。

验收标准：

- 文档和任务列表都把“语义可编辑”和“源码输入不跳顶”列为 P0。
- 有明确复现步骤、预期行为和回归验证方式。
- 后续阶段不再把只读 rich render 视为完成语义编辑。

### 阶段 1：修复源码模式输入跳顶

目标：源码模式可以稳定编辑长文档。

主要工作：

- 拆分源码输入路径和语义编辑事务路径。
- 输入时记录 textarea selectionStart、selectionEnd、scrollTop 和外层 source pane scrollTop。
- 避免每个字符输入都触发 ProseMirror state 全量替换；必要时改为 debounce、idle 同步或仅在切换到语义模式时同步。
- 调整 textarea 高度同步，避免先归零高度导致外层滚动容器回顶。
- 把 trailing blank line 处理从每次输入移到保存、代码块插入或模式切换边界。
- 确保 dirty、stats、outline 仍能随源码输入更新。

验收标准：

- 在 500 行以上文档中间输入文字，滚动位置不跳到顶部。
- 在源码模式连续输入中文、英文和换行，光标不丢失。
- 编辑后切换到语义模式，内容一致。
- 保存后的 Markdown 不额外吞掉用户输入末尾空白以外的正文内容。

### 阶段 2：恢复语义模式基础编辑

目标：语义模式重新成为默认写作面。

主要工作：

- 让可见语义区域由 ProseMirror 编辑面承载，而不是隐藏 ProseMirror host。
- 先支持基础 Markdown 编辑：标题、段落、粗体、斜体、行内代码、链接、引用、无序列表、有序列表、代码块。
- 保留 toolbar 命令通过 EditorCore 执行。
- 保证 paste、Enter、Backspace、Tab、Undo、Redo 在基础结构中可用。
- 重新处理富渲染层：基础结构直接由 ProseMirror 样式表现，复杂块先以可编辑源码块或轻量 NodeView 呈现。
- 确保点击语义内容能聚焦编辑器并显示正确选区。

验收标准：

- 在语义模式点击任意普通段落可以输入文字。
- 标题中输入、删除、回车后 Markdown 序列化正确。
- 列表中 Enter、Tab、Shift+Tab 行为可用。
- 语义模式撤销重做不破坏 Markdown。
- 切换源码模式后看到的 Markdown 与语义编辑结果一致。

### 阶段 3：语义编辑与富渲染合并

目标：把现有技术文档渲染能力逐步纳入可编辑体系。

主要工作：

- Task List：以 ProseMirror 可编辑列表或专门 NodeView 支持 checkbox，勾选回写 `- [ ]` / `- [x]`。
- Code Block：保留源码可编辑，叠加语言标签、标题、复制、行号、换行/横滚和 Shiki 高亮。
- Table：先支持基础 pipe table 的整体编辑和安全序列化，再评估单元格级编辑。
- Math：公式块显示预览，但用户可进入源码编辑 TeX；错误时显示可恢复源码。
- Mermaid：默认保留 fenced code source，提供预览切换和错误提示，不让渲染失败破坏原文。
- HTML：先保留源码或受限预览，补充安全策略后再扩大可编辑能力。

验收标准：

- 技术块在语义模式下至少有一条清晰可编辑路径。
- 任何渲染失败都不改变 Markdown 原文。
- task、code、math、Mermaid 经过语义编辑后仍能保存为标准 Markdown。
- 前端测试覆盖 rich render 到 editable block 的关键回归。

### 阶段 4：多标签、资源管理器与派生状态收敛

目标：让文件导航、多标签和文档派生信息稳定跟随当前文档。

主要工作：

- 完成 `.scratch/tab-outline-navigation` 中的待 triage 工作。
- 统一 Tab 状态写入方式，避免直接原地修改对象导致 UI 不刷新。
- 打开资源管理器中的 Markdown 文件时，新标签立即加载正确正文、路径、dirty 状态和派生信息。
- 切换标签页时同步 Markdown、EditorCore、textarea、Outline、stats、technical previews。
- 资源管理器树保持根节点、目录和文件缩进清晰。
- 大纲显示/隐藏作为 UI 状态，不写入 Markdown。

验收标准：

- 点击文件树中的 Markdown 文件，正文立即显示对应内容。
- 多个标签来回切换不会串文档、串 dirty 状态或串大纲。
- 保存后当前标签标题和状态同步更新。
- Outline 始终来自当前活动标签的 Markdown。

### 阶段 5：MarkdownBridge 与保真等级增强

目标：降低 Markdown round-trip 风险。

主要工作：

- 把 front matter、raw block、code fence meta、table alignment、task list、math、Mermaid 纳入 MarkdownBridge 统一策略。
- 建立 Markdown fixture 套件，覆盖常见技术文档和中文文档。
- 定义保真等级：语义保真、结构保真、格式保真分别承诺什么。
- 为未来 source map 预留数据结构，用于语义节点定位源码行列。
- 明确哪些语法先保留源码，不尝试富编辑。

验收标准：

- 常见 Markdown 结构往返后不丢语义。
- front matter、代码块语言和标题、Mermaid、数学公式不会被误删或改写。
- 文档明确说明早期版本会规范化哪些格式。

### 阶段 6：桌面稳定性与 Windows-first 验收

目标：把编辑闭环变成可日常使用的桌面体验。

主要工作：

- 复核 Tauri 读写、最近文件、快照、窗口状态、菜单命令和拖放打开。
- 保存前创建快照，保存失败保留恢复副本。
- 外部文件修改、删除、只读、权限不足时给出明确状态。
- 大文件使用只读源码模式或延迟语义解析，避免 UI 阻塞。
- 补齐 Windows 路径、中文文件名、空格路径、快捷键和系统主题验证。

验收标准：

- 打开、编辑、保存、另存为、重新打开全流程稳定。
- 保存失败不会丢失当前编辑内容。
- Windows 原生菜单和快捷键能触发同一套 EditorCore / Storage 行为。
- 大文件不会导致窗口长时间无响应。

### 阶段 7：长期扩展与内核探索

目标：在稳定主线之外探索更高保真和自研内核可能性。

主要工作：

- 记录 EditorCore API 在实际功能中的压力点。
- 探索 Markdown source map、局部解析、局部渲染和 raw block 保留。
- 评估 DOMD 式 Markdown-native 内核实验，但不得影响 ProseMirror 主线。
- 在 EditorCore 命令和选区模型稳定后，再评估 AI 写作、插件协议、批注和高级导出。

验收标准：

- 实验能力与稳定编辑路径隔离。
- 新能力必须先适配 EditorCore，不绕过主边界。
- 每个长期方向都有可回退策略。

## Immediate Backlog

1. 新增“源码模式输入不跳顶”回归用例和手动验证脚本。
2. 调整源码输入同步策略，保留 selection 和 scroll。
3. 去掉源码输入时每次调用 trailing blank line normalize 的行为。
4. 让 ProseMirror host 成为语义模式可见、可聚焦、可编辑的主内容面。
5. 暂时降低 richMarkdown 派生渲染层优先级，只保留可安全回写的交互。
6. 为语义模式基础输入补齐手动和自动测试。
7. triage `.scratch/tab-outline-navigation`，决定是否并入阶段 4。

## Out of Scope

- 第一轮不做 AI 写作、云同步、账户、协作、插件市场。
- 第一轮不承诺完整 Typora 功能 parity。
- 第一轮不做完整自研编辑器内核替换。
- 第一轮不做复杂表格拖拽、复杂导出和跨文档知识库。
- 第一轮不把 SQLite 作为 Markdown 正文主存储。

## Further Notes

当前最重要的产品判断是：渲染能力不能替代编辑能力。Task List、表格、代码高亮、数学公式和 Mermaid 都有价值，但它们必须服务于可编辑 Markdown 主线。只要语义模式不可写，项目就偏离了“轻量 Markdown 编辑器”的核心定位。
